1. 프로젝트 개요

Tokyo Last Train Finder는 도쿄 수도권의 전철/지하철 막차 경로를 실시간으로 검색하는 웹 서비스입니다. ODPT(공공교통오픈데이터) API에서 전체 시간표를 캐싱하고, Reverse RAPTOR 알고리즘으로 최적 막차 경로를 탐색합니다.

2. Reverse RAPTOR 알고리즘

2.1 왜 Reverse RAPTOR인가?

일반적인 경로 탐색(Dijkstra, A*)은 "최단 시간"을 찾습니다. 하지만 막차 문제는 "가장 늦게 출발해서 도착할 수 있는 경로"를 찾아야 합니다. 이를 위해 RAPTOR(Round-bAsed Public Transit Optimized Router) 알고리즘을 역방향으로 적용합니다.

핵심 아이디어: 도착역에서 출발하여 거꾸로 탐색합니다. "이 역에서 도착역까지 갈 수 있는 가장 늦은 출발 시각"을 각 역마다 갱신해 나갑니다.

2.2 알고리즘 단계별 동작

초기화

bestByStation[도착역] = (23:59, 빈 여정)
markedStations = {도착역}

메인 루프 (최대 4라운드 = 최대 3회 환승)

Phase 1: 열차 역방향 스캔

  1. marked된 각 역에 대해, 해당 역을 지나는 모든 열차 시간표 조회
  2. 각 열차에서 해당 역 이전의 모든 정차역에 대해:
  3. 출발 시각이 현재 best보다 늦으면 갱신
  4. 새로운 Leg(탑승 구간)를 생성하여 여정에 추가
// 예시: C역이 marked, 열차 A(23:30)→B(23:45)→C(00:00)
// → A역의 best를 23:30으로, B역의 best를 23:45로 갱신
for (각 열차 in C역을 지나는 열차들) {
    arrivalAtC = 열차의 C역 도착시각;
    if (arrivalAtC > bestByStation[C].latestDeparture) continue; // 탑승 불가

    for (C역 이전의 각 정차역 S) {
        depTime = 열차의 S역 출발시각;
        if (depTime > bestByStation[S].latestDeparture) {
            bestByStation[S] = (depTime, S→C Leg + 기존여정);
            newMarked.add(S);
        }
    }
}

Phase 2: 환승 확장

  1. 갱신된 역에서 환승 가능한 역 탐색 (환승 그래프)
  2. 도보 환승 시간 5분 감산
  3. 환승 후 역의 best가 개선되면 TRANSFER Leg 추가
// B역(긴자선)에서 B역(히비야선)으로 환승
afterWalk = bestByStation[B_긴자].latestDeparture - 5분;
if (afterWalk > bestByStation[B_히비야].latestDeparture) {
    bestByStation[B_히비야] = (afterWalk, TRANSFER Leg + 기존여정);
}

Phase 3: 도달 확인

  • 출발역에 도달했으면 결과에 추가
  • 더 이상 갱신이 없으면 종료

결과 처리

  • 동일 경로 구간 조합의 중복 제거 (같은 노선 조합이면 더 늦은 출발만 유지)
  • 출발 시간 내림차순 정렬

2.3 데이터 구조

record Journey(String fromStation, String toStation, List<Leg> legs) {
    LocalTime departureTime()  // 첫 번째 leg의 출발 시각
    LocalTime arrivalTime()    // 마지막 leg의 도착 시각
    int transferCount()        // "TRANSFER" leg 개수
}

record Leg(
    String fromStation, String toStation,
    String railway,           // "TRANSFER"이면 환승 도보
    String railDirection, String trainType, String trainNumber,
    LocalTime departureTime, LocalTime arrivalTime
)

record TimeAndJourney(LocalTime latestDeparture, Journey journey)

2.4 시간/공간 복잡도

항목복잡도실측치 (도쿄 전체)
라운드당 탐색O(S × T × K)S=300+역, T=평균5열차, K=평균20정차
환승 확장O(S × M)M=평균3개 환승역
전체O(R × (S×T×K + S×M))R=4라운드, 50~500ms
공간O(S + T×K + S²)~300~500MB (전체 캐시)

4. 백엔드 상세 설계

4.1 TransitDataCache - 인메모리 캐시 전략

서버 시작 시 ODPT dump API로 도쿄 수도권 전체 교통 데이터를 한 번에 로딩합니다. 이후 모든 조회는 메모리에서 수행하여 쿼리당 API 호출 0회를 달성합니다.

캐시 데이터 구조 (12개 ConcurrentHashMap)

맵KeyValue용도
stationsById역 IDOdptStation역 정보 조회
railwaysById노선 IDOdptRailway노선 정보 조회
trainTimetables시간표 IDOdptTrainTimetable열차별 정차역+시각
fares"출발역::도착역"OdptRailwayFare요금 조회
nameIndex소문자 역명List<역ID>역명 검색
stationToTrainTimetables역 IDList<시간표ID>역별 열차 빠른 조회
transferGraph역 IDSet<역ID>환승 가능 역 탐색
calendarsById캘린더 IDOdptCalendar요일/휴일 판별
왜 ConcurrentHashMap? Spring MVC는 멀티스레드로 요청을 처리하므로, 초기화 이후 읽기 전용이더라도 thread-safe 컬렉션을 사용하여 안전성을 보장합니다.

4.2 캘린더 해석 (resolveCalendar)

1. 특정 날짜 캘린더 확인 (holidays 등에 날짜가 직접 지정된 경우)
2. 요일 기반 판별:
   - 토요일 → SaturdayHoliday (없으면 Holiday로 fallback)
   - 일요일 → Holiday
   - 평일 → Weekday

Fallback 전략

해석된 캘린더로 경로가 0건이면 다른 캘린더로 자동 재시도:

Weekday → SaturdayHoliday → Holiday (순서대로 시도)

이는 ODPT 데이터의 캘린더 정의가 운영사마다 다른 문제를 해결합니다.

4.3 요금 계산 (calculateFare)

총요금 = 0
for (각 탑승 구간 leg) {
    fare = cache.getFare(leg.출발역, leg.도착역);
    if (fare.icCardFare != null) 총요금 += fare.icCardFare;    // IC카드 우선
    else if (fare.ticketFare != null) 총요금 += fare.ticketFare; // 일반 승차권 fallback
}
return 총요금;
IC카드 우선: 도쿄에서는 대부분 Suica/PASMO를 사용하므로 IC카드 요금을 기본으로 표시합니다.

4.4 환승 그래프 구축

  1. Primary: ODPT 데이터의 connectingStation 필드 활용 (공식 환승역)
  2. Secondary: 같은 일본어 역명을 가진 다른 노선의 역을 자동 연결
  3. 양방향(bidirectional) 간선으로 구축

4.5 역명 검색 인덱스

3가지 키로 인덱싱:

  1. 일본어 역명 소문자 (예: 渋谷)
  2. 영어 역명 소문자 (예: shibuya)
  3. 역 ID의 마지막 부분 소문자 (예: shibuya from odpt.Station:...Shibuya)

검색 시 정확 일치 → 부분 일치 순서로 탐색하여 결과 품질을 높입니다.
