1. 프로젝트 개요

1.1 목적

Tokyo Last Train Finder는 도쿄 수도권의 지하철 · JR · 사철 노선망을 대상으로, 임의의 출발역에서 도착역까지 가장 늦게 출발할 수 있는 막차 경로를 탐색해주는 풀스택 웹 애플리케이션이다. 직통 경로뿐 아니라 최대 3회 환승까지의 경로를 모두 도출하며, 환승 시 도보 이동 시간(5분)과 IC카드 기준 구간 요금까지 함께 계산한다.

1.2 타깃 사용자

1차 타깃은 도쿄 권역에서 늦은 시간 귀가하는 일본인 통근자이며, 일본어 · 영어 · 한국어 3개 언어로 자동완성과 UI를 제공한다. 역명 검색은 일본어(한자/히라가나), 영어 로마자, 한국어 별칭(별도 사전) 모두로 가능하다.

1.3 핵심 기능

  • 막차 경로 탐색 — 직통 ~ 3회 환승, 가장 늦은 출발 시각 우선 정렬
  • 역 자동완성 — 300ms 디바운스 + AbortController 적용, 다국어 인덱스 검색
  • 상세 경로 정보 — 출발/도착 시각, 노선, 열차 종별, 환승역, 도보 시간
  • 요금 계산 — IC카드 우선, 사업자 변경 시 구간별 합산
  • 캘린더 자동 판별 — 평일 / 토요일 / 공휴일 시간표를 날짜 기반으로 자동 선택, Fallback 포함

2. 시스템 아키텍처

아래 다이어그램은 개발자 커밋부터 일본인 사용자에게 응답이 전달되기까지의 전체 흐름이다. Jenkins(로컬 Docker)가 GitHub에서 코드를 받아 빌드하고, AWS EC2(도쿄 리전)로 배포한다. EC2 안에서는 Nginx가 정적 파일과 API 프록시를 담당하고, Spring Boot 앱이 인메모리 캐시에 ODPT API 데이터를 통째로 올려두고 응답한다 — 별도의 RDB는 사용하지 않는다.

그림 2-1. 빌드 → 배포 → 런타임 전체 흐름. Jenkins는 사용자 로컬 노트북의 Docker 컨테이너로 동작하며, 빌드 결과를 EC2로 SCP/SSH 전송한다.

2.1 의도적으로 채택하지 않은 컴포넌트

의외로 보일 수 있어 명시한다 — 이 시스템은 다음을 의도적으로 사용하지 않는다.

  • RDB / NoSQL — ODPT 데이터는 하루 단위로도 거의 변하지 않고 전체 크기가 JVM 힙에 들어가므로, DB 도입은 운영 복잡도만 늘린다. 대신 TransitDataCache가 7개 인덱스를 ConcurrentHashMap으로 유지한다.
  • Redis 등 외부 캐시 — 단일 인스턴스 + 인메모리로 충분. 분산이 필요해지면 그때 도입한다.
  • 로그인 / 사용자 데이터 — 막차 검색은 anonymous 단발 조회이므로 인증 계층이 불필요.
  • 로드 밸런서 — 단일 EC2 인스턴스로 충분한 트래픽 규모.
설계 원칙. "데이터는 변하지 않고 전체가 작다"는 도메인 특성을 살려서, 일반적인 백엔드의 DB 계층을 통째로 인메모리 캐시로 대체했다. 그 결과 응답 지연이 ms 단위가 되고, ODPT API에 매 요청마다 의존하지 않게 된다.

3. 기술 스택

HTTP Client Spring WebFlux (WebClient) + Reactor Netty ODPT Dump API는 단일 응답이 큰 JSON 파일(수백 MB). 256MB 버퍼 / 3분 read timeout 설정이 필요해서 RestTemplate보다 WebClient 채택.

Deploy Jenkins (Docker, 로컬) → SCP/SSH → EC2 GitHub Actions 대신 Jenkins를 선택한 이유: 사용자가 Jenkinsfile 기반 파이프라인을 학습하고자 함. Jenkins를 클라우드에 두지 않고 로컬 노트북에 둬서 비용 0.
Runtime EC2 (Amazon Linux 2023, t3.small) + systemd + Nginx 도쿄 리전(ap-northeast-1)으로 일본 사용자 레이턴시 최소화. systemd로 OOM/장애 시 자동 재시작.

5. 패키지 · 클래스 설계 근거

"왜 이렇게 나눴는가"를 패키지별로 정리한다. 일반적인 Spring 레이어 분리에서 파생된 구조이지만, 이 프로젝트 특유의 결정(service와 service.impl 분리, model과 dto 분리, cache 패키지 별도화 등)에 대한 이유를 명시한다.

5.1 controller — 얇게 유지

컨트롤러는 HTTP 진입점만 담당하고, 모든 도메인 로직은 service로 위임한다. 두 컨트롤러 모두 30줄 이내이며 검증/쿼리 파라미터 추출만 한다. 응답은 DTO로 직렬화되어 나간다.

5.2 service / service.impl — 인터페이스 분리

StationSearchService, LastTrainService는 인터페이스이고 구현은 service.impl 패키지에 둔다. 분리한 이유:

  • 테스트 시 mock 구현체 주입 용이 — 컨트롤러 단위 테스트에서 인터페이스 기반으로 stubbing.
  • 구현 교체 가능성 확보 — 예를 들어 향후 RAPTOR 대신 다른 알고리즘을 시험할 때 인터페이스만 유지하면 된다.
  • 패키지 가시성 관리 — ReverseRaptorEngine은 외부에서 직접 호출하지 않고 LastTrainServiceImpl이 위임하는 내부 컴포넌트. impl 안에 둠으로써 응집도 표현.

5.3 model vs dto — 입력 객체와 출력 객체의 분리

model/

외부에서 들어온 데이터의 형태. ODPT API JSON을 그대로 매핑하는 Java Record. RDF 표기(owl:sameAs, dc:title, geo:lat)를 그대로 받는 것이 목적이라 필드명이 ODPT 스키마에 종속.

dto/

외부로 나가는 데이터의 형태. 프론트엔드가 다루기 좋은 평탄화된 구조. 다국어 이름(ja/en/ko)을 한 객체에 모아두고, 환승 정보를 별도 객체로 추출한다.

이 분리는 "외부 API 변경이 우리 응답 스키마에 새지 않도록"이라는 안티-부패 계층(anti-corruption layer) 역할을 한다. ODPT가 필드명을 바꾸면 model/만 수정하면 되고, 프론트엔드 응답은 그대로 유지된다.

5.4 cache — DB 대체 계층

TransitDataCache는 단순한 캐시가 아니라 이 프로젝트의 데이터 계층 그 자체다. 일반 백엔드의 Repository / DAO에 해당하는 역할을 한다. 그래서 service나 repository 패키지에 두지 않고 별도 cache 패키지를 만들어 책임을 명확히 표시했다. 내부에 9개의 ConcurrentHashMap 인덱스(7장에서 상세)와 그것을 빌드하는 메서드를 가진다.

5.5 client — 외부 시스템 어댑터

OdptApiClient는 ODPT API와 통신하는 유일한 클래스. 다른 모든 코드는 cache를 경유하며, ODPT API에 직접 의존하는 부분은 이 한 클래스로 한정된다. 이렇게 함으로써 ODPT가 점검 중이거나 API 명세가 바뀔 때 OdptApiClient와 model/만 손대면 된다.

5.6 config — Bean 정의의 단일 위치

  • AppConfig — WebClient 빈을 정의 (메모리 256MB, connect 30s, read 3min, redirect follow).
  • OdptApiProperties — application.yml의 odpt.api.*를 타입 안전하게 바인딩.
  • WebConfig — CORS 정책 (개발 시 Vite dev 서버:3000 허용).

빈 정의를 산발시키지 않고 config/에 모아두면 의존성/설정의 변화 지점을 한 패키지에서 찾을 수 있다.

5.7 health — 운영 가시성

CacheHealthIndicator는 TransitDataCache.isReady()를 /actuator/health의 응답에 반영한다. 부팅 후 ODPT 데이터를 모두 받아오기까지 약 2~3분이 걸리는데, 그 사이에 트래픽이 들어오면 안 되기 때문에 deploy.sh가 health UP을 polling 해서 무중단 배포의 신호로 사용한다.

6. 데이터 모델 (ODPT 엔티티)

이 프로젝트는 RDB 테이블을 가지지 않는다. 대신 ODPT API에서 받아오는 7종의 엔티티를 Java record로 매핑하고, 이를 인메모리 인덱스에 적재한다. 본 절은 그 7개 엔티티의 필드 명세이며, 다음 7장은 인덱스 구조다.

6.1 OdptStation — 역

필드JSON 키의미
idowl:sameAs역 고유 ID. 예: odpt.Station:TokyoMetro.Ginza.Shibuya
titledc:title일본어 표기 (e.g. 渋谷)
stationTitleodpt:stationTitle{ja, en} 다국어 맵
operatorodpt:operator운영 사업자 ID
railwayodpt:railway소속 노선 ID
stationCodeodpt:stationCode역 코드 (e.g. G-01)
latitudegeo:lat위도
longitudegeo:long경도
connectingRailwayodpt:connectingRailway환승 가능 노선 ID 목록
connectingStationodpt:connectingStation환승 가능 역 ID 목록 (transferGraph의 1차 소스)

6.2 OdptRailway — 노선

필드의미
id, title, railwayTitle노선 ID, 일본어 표기, 다국어 맵
operator, lineCode, color운영자, 노선 코드(e.g. G), 색상
stationOrder역 순서 배열 ({index, station}) — railwayStationOrder 인덱스의 소스

6.3 OdptTrainTimetable — 열차 시간표 (가장 중요)

한 대의 열차가 출발역에서 종점까지 정차하는 모든 역과 시각을 담은 객체. RAPTOR의 핵심 입력.

필드의미
id고유 ID (e.g. odpt.TrainTimetable:TokyoMetro.Ginza.A1234)
railway운행 노선 ID
railDirection방향 ID (상행/하행)
calendar운행 캘린더 ID (Weekday/SaturdayHoliday/Holiday)
trainNumber열차 번호
trainType열차 종별 ID (각정/쾌속/특급 등)
originStation, destinationStation시발역, 종점역
stops핵심. List<TrainStop> — 정차역마다 departureTime, arrivalTime, platformNumber를 가진다.

TrainStop은 출발/도착 시각이 둘 다 있는 경우(중간역) / 하나만 있는 경우(시발/종점) 모두 처리해야 하므로, Record 안에 effectiveTime(), effectiveStation() 메서드를 두어 추상화한다.

6.4 OdptStationTimetable — 역 기준 시간표

특정 역에서 특정 방향, 특정 캘린더에 출발하는 모든 열차를 모아둔 객체. RAPTOR에서는 거의 사용하지 않고, 역별 디스플레이 용도. (stationId, railDirection, calendarId)로 식별.

6.5 OdptCalendar — 운행 캘린더

Weekday, SaturdayHoliday, Holiday 등 미리 정의된 캘린더와, 특정 날짜를 명시적으로 매핑하는 days 배열. resolveCalendar(LocalDate)는 days 매칭을 우선하고, 없으면 요일 기반으로 fall through.

6.6 OdptRailwayFare — 구간 요금

(fromStation, toStation) 쌍에 대한 IC카드 요금 / 종이 티켓 요금 / 어린이 요금. icCardFare를 우선 사용하고, 없으면 ticketFare로 fallback.

6.7 OdptTrainType — 열차 종별

각 정거(Local), 쾌속(Rapid), 특급(Express) 등 종별 분류. ID와 다국어 표기 맵을 가진다.

7. 인메모리 캐시 인덱스 구조

TransitDataCache는 위 7개 엔티티를 9개의 인덱스로 변형하여 보관한다. 이 인덱스들이 RDB의 인덱스/조인을 대체하며, 모든 조회는 O(1) 또는 O(log n)으로 수행된다.

인덱스타입 / 키 → 값역할 · 사용처
stationsById Map<String, OdptStation>
stationId → Station		 역 ID로 역 정보 조회. 모든 응답 변환의 시작점.
railwaysById Map<String, OdptRailway> 노선 ID → 노선 (이름, 색상, 운영자).
trainTypesById Map<String, OdptTrainType> 열차 종별 ID → 다국어 표기.
calendarsById Map<String, OdptCalendar> resolveCalendar(date)가 사용. days 매칭 우선.
stationTimetables Map<String, OdptStationTimetable>
"stationId::direction::calendar" → SST		 역 기준 시간표 조회 (디스플레이 용도). 합성 키 사용.
trainTimetables Map<String, OdptTrainTimetable>
trainTimetableId → TT		 RAPTOR가 열차 ID로 시간표 본체를 가져올 때 사용.
fares Map<String, OdptRailwayFare>
"from::to" → Fare		 요금 조회. 사업자 변경 시 leg별로 합산.
nameIndex Map<String, List<String>>
소문자 역명 → stationId[]		 자동완성 핵심. 일본어/영어/한국어 별칭/ID 후미를 모두 키로 등록.
stationToTrainTimetables Map<String, List<String>>
stationId → trainTimetableId[]		 RAPTOR 성능 핵심. "이 역을 지나는 열차들"을 풀스캔 없이 조회. 없으면 매 라운드 O(N) 풀스캔이 발생.
transferGraph Map<String, Set<String>>
stationId → 환승 가능 stationId[]		 RAPTOR 환승 확장 핵심. connectingStation 1차, 같은 이름 그룹핑이 2차 소스. 양방향 등록.
railwayStationOrder Map<String, List<String>>
railwayId → 정렬된 stationId[]		 노선 내 역 순서 표현. OdptRailway.stationOrder를 index로 정렬한 결과.

7.1 인덱스 빌드 순서

refresh()                    // @PostConstruct에서 호출
├── loadStations()           // stationsById
├── loadRailways()           // railwaysById, railwayStationOrder
├── loadTrainTypes()         // trainTypesById
├── loadCalendars()          // calendarsById
├── loadStationTimetables()  // stationTimetables
├── loadTrainTimetables()    // trainTimetables
├── loadFares()              // fares
├── buildNameIndex()         // nameIndex (다국어 + alias dict)
├── buildTransferGraph()     // transferGraph (양방향 + 동명역 그룹)
└── buildStationTrainTimetableIndex()  // stationToTrainTimetables

빌드 순서가 중요한 이유: buildTransferGraph()는 stationsById가 채워진 뒤에 실행되어야 하고, buildStationTrainTimetableIndex()는 trainTimetables 다음에 실행되어야 한다. 리프레시는 synchronized 메서드로 보호되며, 그 동안 ready 플래그가 false가 되어 /actuator/health가 DOWN을 반환한다.

주의. JVM 힙은 -Xmx 2g로 설정되어 있다. ODPT 전체 dump를 메모리에 올리는데, 역 약 5천 + 시간표 수만 건 + 요금 수만 건 규모라 t3.small(2GB)에서는 마진이 빠듯하다. 힙이 부족할 경우 t3.medium(4GB)으로 인스턴스 타입 업그레이드가 필요하다.

8. 핵심 알고리즘: Reverse RAPTOR

일반적인 RAPTOR(Round-bAsed Public Transit Optimized Router)는 출발역에서 도착역 방향으로 라운드별 확장하며 "가장 빠른 도착 시각"을 찾는 알고리즘이다. 이 프로젝트의 Reverse RAPTOR는 같은 알고리즘을 시간 축을 뒤집어서 적용한다 — 도착역에서 시작해서 거꾸로 확장하며 "가장 늦은 출발 시각"을 찾는다.

8.1 라운드 정의

라운드의미
Round 0도착역에 직통으로 가는 열차만 고려 (환승 0회)
Round 1Round 0 결과 + 환승 1회로 도달 가능한 역
Round 2+ 환승 2회
Round 3+ 환승 3회 (최대)

8.2 각 라운드 내부 처리

  1. 열차 역방향 스캔 — 이번 라운드의 marked station들 각각에 대해, 그 역을 지나는 모든 OdptTrainTimetable을 stationToTrainTimetables 인덱스로 조회한다. 그 열차의 정차역들 중 marked station보다 앞쪽(시간 축 이전) 역들에 대해, 해당 역에서의 출발 시각이 기존 best보다 늦으면 갱신.
  2. 환승 전파 — 갱신된 역에서 transferGraph로 연결된 환승역들에 대해, 환승 도보 시간 5분을 감산한 시각으로 best를 시도. 더 늦은 시각이면 갱신.
  3. 출발역 도달 체크 — fromStation의 best가 갱신되어 있으면, 그 시점의 journey를 결과 배열에 추가. (라운드별 최적 경로가 결과에 누적된다.)

8.3 핵심 자료구조

Map<String, TimeAndJourney> bestByStation
   // stationId → (latestDeparture, journey)
   // 해당 역에서 도착역까지 갈 수 있는 가장 늦은 출발 시각

record TimeAndJourney(LocalTime latestDeparture, Journey journey) {}
record Journey(String fromStation, String toStation, List<Leg> legs) {}
record Leg(String fromStation, String toStation,
            String railway, String railDirection,
            String trainType, String trainNumber,
            LocalTime departureTime, LocalTime arrivalTime) {
  // railway == "TRANSFER" 이면 환승 leg
}

8.4 심야 시간 처리

일본 철도 시간표는 자정을 넘는 열차를 24:30, 25:00처럼 표기한다. parseTime()이 hour >= 24인 경우 24를 빼서 LocalTime으로 변환한다 (예: 25:30 → 01:30). 그리고 LAST_TRAIN_CUTOFF = 01:30을 기준으로, arrivalTime < 01:30이면 새벽 도착으로 간주하여 23:59 이후 열차로도 탑승 가능 처리한다 (isAfterMidnight).

8.5 캘린더 Fallback

LastTrainServiceImpl.resolveCalendarWithFallback()은 오늘 날짜로 캘린더를 결정한 뒤, 그 캘린더로 검색해서 결과가 비어있으면 다른 캘린더(Weekday → SaturdayHoliday → Holiday)를 차례로 시도한다. 이는 일부 노선이 특정 캘린더의 데이터를 ODPT에 제공하지 않는 경우를 우회하기 위한 처리다.

8.6 결과 후처리

  • 중복 제거 — 같은 leg 시퀀스(railway × from × to)는 하나만 남기고, 그 중 출발 시각이 가장 늦은 것을 선택.
  • 정렬 — 출발 시각 내림차순. 사용자에게 "가장 늦게 출발할 수 있는" 경로가 맨 위에 보인다.
  • 요금 계산 — LastTrainServiceImpl.calculateFare()가 ride leg마다 fare 인덱스를 조회해 IC카드 요금을 합산.

10. 배포 파이프라인

Jenkins(로컬 Docker) → GitHub → 빌드 → SCP → systemd 재시작 → health 확인의 순서로 동작한다. 아래는 Jenkinsfile이 정의하는 stage 흐름이다.

5. DeployJAR + deploy.sh + frontend/dist/를 EC2로 SCP, SSH로 deploy.sh 실행SCP, SSH (key from Jenkins credentials)

10.1 Jenkins 자격증명 이름

Jenkinsfile은 다음 credential ID를 참조한다 — Jenkins UI에서 동일한 이름으로 등록되어 있어야 한다.

  • ec2-host — EC2 퍼블릭 IP (Secret Text)
  • ec2-ssh-key — EC2 SSH 프라이빗 키 (SSH Username with private key, username ec2-user)
  • odpt-api-key — ODPT 컨슈머 키 (Secret Text)

10.2 무중단 배포 보장

JAR 교체는 app.jar.new를 SCP로 받은 뒤 mv로 atomic 치환한다. systemctl restart 직후에 새 프로세스의 캐시 로딩(2~3분)이 끝날 때까지 health가 DOWN으로 머물고, deploy.sh가 그동안 polling하여 UP을 확인한 후 종료한다 — 즉, 빌드 잡의 종료가 곧 "사용자에게 응답 가능 상태" 시점이다. 실패 시 journalctl 마지막 50줄을 출력하고 exit 1.

10.3 시퀀스 다이어그램

그림 10-1. 빌드~헬스체크 시퀀스. systemd → 캐시 로딩(2~3분) → UP 확인 후 잡 종료.

[Service]

Restart=on-failure
RestartSec=10
StandardOutput=journal

11.3 Nginx 구성

SPA 라우팅(try_files) + API 프록시 + Actuator 외부 차단 + gzip. 

server {

    server_name _;
    root /var/www/tokyo-last-train;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;     # SPA 라우팅

    location /api/ {
        proxy_pass http://localhost:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;

11.5 보안 / 비밀 관리

  • ODPT API 키는 /opt/tokyo-last-train/.env에 두고 systemd EnvironmentFile=로 주입. 코드/리포지터리에는 절대 들어가지 않는다.
  • Jenkins → EC2 SSH 접속은 ed25519 전용 키(jenkins-deploy-key) 사용, 비밀번호 없음.
  • Actuator는 외부 노출되지 않는다 (Nginx에서 403 반환).
  • HTTPS는 Let's Encrypt(scripts/enable-https.sh)로 도메인 발급 후 활성화 예정.

12. 향후 과제

  • 도메인 등록 + HTTPS — Cloudflare/Porkbun에서 도메인 구입 → DNS A 레코드 → scripts/enable-https.sh로 Let's Encrypt 발급.
  • Search Console 등록 — DNS TXT 레코드 검증 + sitemap 제출 → 일본 구글에서 인덱싱.
  • 인스턴스 업그레이드 또는 Elastic IP — 캐시 메모리 압박이 있으면 t3.medium으로, IP 안정성을 위해 Elastic IP 부착.
  • JR 외 사철 데이터 보완 — ODPT가 제공하지 않는 일부 사철은 connectingStation이 없어 환승 그래프가 불완전. 동명역 그룹핑이 fallback이지만 정확도 한계.
  • 모니터링 — 현재는 journalctl만 의존. CloudWatch 또는 Grafana Cloud로 메트릭/알람 보강.
  • 주기 캐시 갱신 — TransitDataRefreshScheduler가 존재하지만 트리거 주기를 운영 중 검증 필요. ODPT는 일 단위 변경이므로 매일 새벽 1회로 충분.
  • RAPTOR 최적화 — 현재 scanRoutesReverse는 라운드별 marked station × 그 역의 시간표 수만큼 반복. 시간표 수가 많은 환승역(신주쿠/도쿄)에서 라운드 1~2 비용이 큼. trip-by-trip pruning 도입 여지.
